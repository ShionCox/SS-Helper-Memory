# 结构化 Capture

## Capture 做什么

Capture 把当前聊天、角色卡和世界书整理成人物、事件、观察和事实。模型只提供候选内容，代码才决定哪些内容可以写入。

输入分成三部分：

- `allowedSourceRefs`：本次允许引用的来源 ID。
- `existingMemoryContext`：只用于消歧和去重，不能当作新事实证据。
- `sourceBlocks`：可以成为证据的来源正文。

模型必须返回一个 JSON 根对象，其中固定包含 `actorCandidates`、`episodes`、`observations`、`facts` 四个数组。

## 代码怎样检查

代码逐条检查字段、枚举、来源引用、证据原文和局部 ID。合法项立即写入；非法项不会拖垮同批的合法项，也不会被悄悄改成别的类别。

例如，未知 `kind` 不会自动变成 `other`。只有模型明确输出合法的 `other` 才会接受。

## 什么是部分完成

同一批中有合法项和失败项时，任务显示“部分完成”：

- 已接受：已经通过校验并写入。
- 待处理：保留错误码、字段位置、来源和候选快照，尚未写入。

整个 JSON 无法解析时，本批失败，不会尝试拼接多个 JSON。

## 怎样处理失败项

打开记忆工作台的“审计记录”，展开 Capture 失败项：

1. 勾选待处理项。
2. 选择“定向修复”或“忽略所选”。
3. 修复按人物、事件、观察、事实分组，每种类型最多请求一次；界面会显示预计请求数。

修复结果仍会经过相同的代码校验。通过后直接写入；再次失败时保留记录和尝试次数，可以以后再处理。忽略只改变审计状态，不会写入记忆。

## 哪些错误会自动重试

只有超时、限流、网络错误和明确的临时 Provider 故障会自动重试一次。认证失败、JSON 格式错误、Schema 错误、来源引用错误和语义错误不会自动重试。

## 新增字段时要同步什么

新增或修改 Capture 字段时，必须同时检查：

1. Capture 提示词和 JSON 骨架。
2. 解析后的逐条校验。
3. 写入映射和审计快照白名单。
4. 定向修复输入。
5. 对应测试和本页说明。

不要只修改提示词或 Schema。
